# Milk Frog 项目规范

## 概述

在 `milk-frog` 仓库内改代码时使用这个 skill。它总结了当前项目的实际结构、开发方式，以及首页视频页的实现约束，避免按通用 Web 项目的习惯误改。

## 项目结构

- 这是一个微信小程序仓库，不是普通 Node Web 项目。
- 根目录应用文件包括 `app.js`、`app.json`、`app.wxss`、`project.config.json` 和 `sitemap.json`。
- 当前只有一个活跃页面：`pages/index/`，包含 `index.js`、`index.wxml`、`index.wxss`、`index.json`。
- 可复用的纯逻辑放在 `utils/`。
- 静态图片资源放在 `assets/`。

## 开发规则

- 使用微信小程序模式：`Page({})`、`setData()`、WXML 事件绑定和 WXSS。
- 页面文件尽量只处理生命周期、事件响应和界面状态更新。
- 与微信运行时无关、适合复用或测试的逻辑放进 `utils/`。
- 遵循仓库现有格式规则：2 空格缩进、分号、双引号、尾随逗号、100 字符换行、CRLF。
- 新增工具页或资源页时，页面主体容器默认需要控制在 `max-width: 375px`，避免真机下内部卡片再次溢出。
- 浅色卡片上的小型操作按钮默认不要额外加边框和阴影，优先通过底色差异区分层级，避免按钮显得发灰发脏。
- 不要默认仓库存在 `npm run` 流程，这个项目主要依赖微信开发者工具调试。
- 这个项目默认不要求编写或运行自动化测试，除非用户明确要求补测试。

## 修改流程

除非任务明确要求，否则按下面顺序处理：

1. 先检查目标页面或工具模块。
2. 判断问题属于 WXML/WXSS、页面状态，还是 `utils/` 纯逻辑。
3. 用最小改动保持现有小程序写法一致。
4. 最后在微信开发者工具里验证页面表现。

## 首页视频说明

当前首页是一个单视频页面，并带有自定义播放/停止控制。

- 关键文件是 `pages/index/index.js`、`pages/index/index.wxml`、`pages/index/index.wxss`。
- 视频加载状态被抽到了 `utils/page-loading-state.js`。
- 当前页面会在 `loadedmetadata` 或 `canplay` 后隐藏加载遮罩，并有 8 秒超时兜底。
- 如果视频在模拟器里有声音但没画面，优先检查视频编码和真机表现，不要先改页面逻辑。
- 微信小程序 `<video>` 兼容性更稳的格式优先用 `MP4 + H.264 + AAC`，像素格式尽量是 `yuv420p`。

## 验证方式

- 需要格式化时使用：`npx prettier --write .`
- UI 改动必须在微信开发者工具里验证，因为仓库没有 CLI 构建流水线。
- 如果用户没有明确要求，不主动补自动化测试，也不把“没跑测试”当成阻塞项。

## 常见误区

- 不要假设模拟器和真机在视频播放上表现完全一致。
- 不要在仓库还没引入的前提下擅自加通用 Web 工具链。
- 不要默认每次修改都要补 `tests/`，这个项目以页面调试和手动验证为主。
- 不要把可复用状态逻辑散落在页面里，能抽到 `utils/` 的就抽出去。
